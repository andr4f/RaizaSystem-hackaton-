Guía para el equipo — qué hace cada componente y cómo se conecta todo

1. El reto y cómo lo cubrimos

El reto pide un sistema que conecte la trazabilidad agrícola de exportación con la experiencia turística y el comprador internacional, bajo los estándares Fairtrade y Rainforest Alliance. Se estructura en 3 pilares:

Pilar del retoQué exigeCómo lo cubre el backend
1. Trazabilidad para exportación Registrar de forma transparente, inmutable y verificable el origen, condiciones de cultivo y cumplimiento de sellos. ✓ Cubierto — módulos producer, farm, lot, certification + cadena de hash SHA-256 en trace_event
2. Vinculación turística Un turista escanea un código QR, conoce la historia del agricultor, verifica el cultivo y adquiere el producto. ✓ Cubierto — módulo tourism + vista pública QR sin login
3. Cadena de valor unificada Flujo digital continuo: productor → operador turístico → exportador → comprador. ✓ Cubierto — módulos lead, exporter conectan los 4 actores

⚠ Brecha frente a la rúbrica: La Innovación (20%) exige IA como componente central — el documento dice textualmente que "la ausencia de IA como componente central limita este criterio a máximo 50 puntos". Hoy el backend no tiene ningún endpoint de IA. Ver sección 9.

2. Arquitectura general — monolito modular en capas

Una sola aplicación Spring Boot, organizada por módulos de dominio. Cada petición atraviesa estas capas:

CLIENTE — React (dashboard interno) + vista pública QR (móvil)
▼ HTTP / REST JSON · Bearer JWT
SEGURIDAD — JwtAuthenticationFilter valida el token y carga el usuario · SecurityConfig define rutas públicas vs protegidas
▼
CONTROLLER (api/) — recibe la petición, valida con @Valid, aplica @PreAuthorize por rol, devuelve ApiResponse
▼
SERVICE — la lógica de negocio: reglas, transacciones, dispara eventos de trazabilidad
▼
REPOSITORY (JPA) — acceso a datos · MAPPER (MapStruct) — convierte entidad ↔ DTO
▼ JDBC
PostgreSQL — esquema gestionado por Flyway · 16 tablas

Regla clave: el controller nunca expone entidades, solo DTOs. La lógica vive en el service, no en el controller.

4. El flujo completo de punta a punta

Así se conectan los 4 actores en una sola cadena digital. Las etiquetas EVENTO son eventos de trazabilidad que el sistema graba automáticamente.

① PRODUCTOR — genera el producto
Registra productor y finca (origen territorial)
Crea lote de café/banano/cacao → nace en CERTIFICATION_PENDING  LOT_CREATED
Registra eventos del cultivo  HARVEST_COMPLETED QUALITY_CHECKED
Sube evidencia de certificación (Fairtrade / Rainforest Alliance) → PENDING_VALIDATION
▼
② ADMIN — valida el cumplimiento
Revisa y valida la certificación → VALIDATED  CERTIFICATION_VALIDATED
El lote pasa automáticamente a AVAILABLE (ya es exportable y visible)
▼
③ OPERADOR TURÍSTICO — activa la experiencia
Crea una experiencia ("Cata de café en Minca") con su QR único
Vincula el lote certificado a la experiencia  TOURISM_LINKED
▼
④ TURISTA / COMPRADOR — descubre y compra
Escanea el QR en Santa Marta → ve productor, finca, certificaciones y timeline  QR_SCANNED
Deja una intención de compra (lead) → NEW  PURCHASE_INTENT_CREATED
▼
⑤ EXPORTADOR — cierra la oportunidad comercial
Recibe el lead, verifica la trazabilidad y certificaciones validadas
Crea revisión de exportación (incoterm, fecha estimada)  EXPORT_REVIEWED
El lote pasa a IN_EXPORT_REVIEW → al cerrar: SOLD

6. Máquinas de estado

Estado del lote (LotStatus)

CERTIFICATION_PENDING ──► AVAILABLE ──► RESERVED ──► IN_EXPORT_REVIEW ──► SOLD
                                                             └──► INACTIVE

Transiciones automáticas: crear lote → PENDING · admin valida certificación → AVAILABLE · exportador crea revisión → IN_EXPORT_REVIEW.

Estado de validación de certificación

PENDING_VALIDATION ──► CONDITIONALLY_VALID ──► VALIDATED
                  └──► REJECTED          └──► EXPIRED

Regla de negocio clave: el lote NO es exportable hasta que su certificación esté validada. "Registro abierto, comercialización restringida por cumplimiento".

Estado del lead (LeadStatus)

NEW ──► CONTACTED ──► QUALIFIED ──► IN_EXPORT_REVIEW ──► CLOSED_WON
                                               └──► CLOSED_LOST

7. La trazabilidad inmutable — cadena de hash

Cada TraceEvent firma criptográficamente al anterior con SHA-256. Esto es lo que demuestra "inmutable y verificable" ante el jurado (Pilar 1) y aporta el componente tipo blockchain sin la complejidad de una blockchain real.

Evento 1 (LOT_CREATED)
  previousHash = "GENESIS"
  hashValue = SHA256("lotId|LOT_CREATED|fecha|PRODUCER|...|GENESIS") = a3f9c2...

Evento 2 (HARVEST_COMPLETED)
  previousHash = a3f9c2...  ← toma el hash del evento 1
  hashValue = SHA256("...|a3f9c2...") = 7b4e1d...

Evento 3 (QUALITY_CHECKED)
  previousHash = 7b4e1d...  ← encadena al evento 2
  hashValue = SHA256("...|7b4e1d...") = 9c2a8f...

Si alguien altera el evento 2, su hash cambia, lo que rompe el 3, el 4, y toda la cadena. La manipulación es detectable.

9. Qué falta — prioridad para la demo

🔴 CRÍTICO 1 — Migración Flyway desactualizada (bloquea el arranque)

El V1__init_schema.sql NO incluye lo que agregamos después: la tabla certification_application, la columna app_user.onboarding_completed, ni las columnas validation_notes / validated_by / validated_at de lot_certification. Como ddl-auto=validate, la app no arranca contra una BD limpia.

Solución: actualizar V1 (si la BD está vacía) o crear V2__add_certification_application.sql.

🔴 CRÍTICO 2 — Sin endpoints de IA

Innovación pesa 20% y el reto exige IA central. Faltan, según lo que ya definió el equipo:

  • POST /api/v1/ai/certification-guide — guía paso a paso para el campesino (qué certificación necesita y cómo obtenerla)
  • POST /api/v1/ai/onboarding — wizard introductorio según el rol
  • POST /api/v1/ai/validate-certification — pre-validación automática de certificaciones
  • GET /api/v1/lots/{id}/ai-summary — explicación del lote para el comprador

Requiere integrar Anthropic Claude API. Es el mayor diferenciador del pitch.

🟡 IMPORTANTE — Datos de demo (seed)

No hay V2/V3 con datos. Sin seed, el jurado ve pantallas vacías. Se necesita: catálogo de productos y certificaciones, usuarios por rol, y un lote demo con eventos ya cargados.

🟡 OPCIONAL — Endpoint de métricas/dashboard

El criterio de Impacto (20%) pide "indicadores concretos: número de productores, lotes trazados, leads generados". Un endpoint GET /api/v1/dashboard/metrics daría números directos para el pitch.
